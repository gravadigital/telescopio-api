### Arquitectura del Servicio

**Estructura de carpetas relevante:**
```

internal/
```

**Patrones arquitectónicos:**
- Los handlers reciben dependencias por constructor (`NewXxxHandler(repo, cfg)`)
- La lógica de negocio vive en el handler o en un service dedicado, NO en el repositorio
- Los repositorios solo hacen acceso a datos
- Los handlers usan `c.ShouldBindJSON(&req)` para validar input
- Los errores se retornan como `gin.H{"error": "...", "code": "SCREAMING_SNAKE_CASE"}`
- El logger se obtiene via `logger.Handler("nombre")` en el constructor

**Manejo de errores (patrón existente):**

```go
        "error":   "Invalid request payload",
```


### ADRs Relevantes

No hay ADRs documentados en el producto. Las decisiones técnicas se extraen del código existente:

- **JWT:** HS256, expira en 24h, claims: `user_id`, `email`, `role`, issuer `"telescopio-api"`. Generado con `auth.GenerateToken(userID uuid.UUID, email string, role participant.Role) (string, error)`
- **Seguridad de credenciales:** Siempre desde variables de entorno, nunca hardcodeadas. Ver `config.go` como patrón.
- **Validación de token Google:** Debe hacerse contra la API de Google (no solo decodificar localmente) para verificar firma y expiración.

### Contexto de API

**Endpoints existentes de autenticación (patrón a seguir):**

```
POST /api/v1/users                    # Registro tradicional → 201 {token, user}
POST /api/v1/users/authenticate       # Login tradicional   → 200 {message, token, user}
```

**Response de user en endpoints existentes:**
```json
{
  "token": "<jwt>",
  "user": {
    "id": "<uuid>",
    "name": "<string>",
    "lastname": "<string>",
    "email": "<string>",
    "role": "<string>",
    "created_at": "<timestamp>"
  }
}
```

**Nuevos endpoints a implementar:**

`POST /api/v1/auth/google/verify`
- Request: `{"token": "string"}`
- Response 200 (usuario nuevo): `{"status": "new_user", "google_token": "string", "profile": {"email": "string", "suggested_name": "string"}}`
- Response 200 (usuario existente): `{"status": "existing_user", "token": "<jwt>", "user": {"id": "string", "email": "string", "username": "string"}}`
- Error 401: token inválido/expirado
- Error 500: fallo al contactar Google

`POST /api/v1/auth/google/register`
- Request: `{"token": "string", "username": "string"}`
- Response 201: `{"token": "<jwt>", "user": {"id": "string", "email": "string", "username": "string"}}`
- Error 400: payload inválido (token vacío, username vacío)
- Error 401: token inválido/expirado
- Error 409: username ya en uso

**Ambos endpoints son públicos (sin `JWTAuthMiddleware`).**

**Autenticación:** Bearer token en header `Authorization`. El middleware existente `auth.JWTAuthMiddleware()` no se usa en estos endpoints.

### Puntos de Integración

**Google OAuth API (externo):**
- URL: `https://oauth2.googleapis.com/tokeninfo?id_token=<TOKEN>`
- Método: GET
- Respuesta exitosa incluye: `sub` (google_id), `email`, `name`, `email_verified`
- Error: respuesta no-2xx o campo `error_description`
- Autenticación: no requiere credenciales del servidor para validar `id_token`; el `GOOGLE_CLIENT_ID` se usa para verificar que el `aud` del token coincide con nuestra app
- Librería alternativa: `google.golang.org/api/oauth2/v2` o llamada HTTP directa a tokeninfo

**No hay message queues ni servicios internos adicionales involucrados.**

```go
// En interfaz UserRepository (repository.go):
GetByGoogleID(googleID string) (*participant.User, error)

// Implementacion (user_repository.go) — mismo patron que GetByEmail:
func (r *PostgresUserRepository) GetByGoogleID(googleID string) (*participant.User, error) {
    r.log.Debug("retrieving user by google_id", "google_id", googleID)
    if googleID == "" {
        return nil, errors.New("google_id cannot be empty")
    }
    var user participant.User
    if err := r.db.Where("google_id = ?", googleID).First(&user).Error; err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            return nil, errors.New("user not found")
        }
        return nil, fmt.Errorf("failed to get user by google_id: %w", err)
    }
    return &user, nil
}
```

### Tarea 4: Handler `POST /api/v1/auth/google/verify`

**Status:** Completed

#### Descripcion

Crear `GoogleAuthHandler` en `internal/handlers/google_auth_handler.go` con el metodo `VerifyGoogleToken`. Este handler:

1. Parsea y valida el request `{"token": "string"}` (token no vacio)
2. Llama a `verifyGoogleToken(req.Token, h.cfg.Google.ClientID)` — si falla por token invalido: 401; si falla por red: 500
3. Llama a `resolveUser(profile, h.userRepo)`:
   - Si `Status == "existing_user"`: genera JWT con `auth.GenerateToken` y retorna 200 con `{status: "existing_user", token, user}`
   - Si `Status == "new_user"`: retorna 200 con `{status: "new_user", google_token: req.Token, profile: {email, suggested_name}}`

#### Criterios de Aceptacion

1. `POST /api/v1/auth/google/verify` con token vacio retorna 400
2. Con token invalido de Google retorna 401 `{"code": "INVALID_GOOGLE_TOKEN"}`
3. Con error de red hacia Google retorna 500 `{"code": "GOOGLE_API_ERROR"}`
4. Con token valido de usuario nuevo retorna 200 `{"status": "new_user", "google_token": "...", "profile": {"email": "...", "suggested_name": "..."}}`
5. Con token valido de usuario existente (por google_id o por email) retorna 200 `{"status": "existing_user", "token": "<jwt>", "user": {"id": "...", "email": "...", "username": "..."}}`
6. El JWT generado usa `auth.GenerateToken` (mismo claims que flujo tradicional)
7. El campo `username` en la respuesta corresponde al campo `Name` del modelo `User`

#### Detalles Tecnicos

**Referencias al Contexto Arquitectonico:**
- Seguir patron de `UserHandler` en `internal/handlers/user_handler.go`
